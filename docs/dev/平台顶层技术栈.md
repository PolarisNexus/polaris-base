# [PolarisNexus](https://github.com/PolarisNexus) 平台顶层技术栈

> 地基文档 —— 面向 AI 开发参考，不解释原理，原理另建专项文档。
>
> 旨在构建一个**基于多语言微服务（Polyglot Microservices）的企业级技术底座**。

---

## 1 架构概述

多语言微服务 + 可组合架构。用成熟开源组件拼装平台基座，业务以独立产品接入。

**一个平台基座 + 多产品矩阵**

- **平台基座:** 统一接管 IAM、网关、多租户路由、可观测性、容器运行环境。
- **产品接入:** Java 业务群、Python AI 群及其他语言模块，通过 OpenAPI 规范、HTTP/gRPC 协议对接，独立部署。

### 架构分层

| 层 | 职责 | 核心组件 |
|---|---|---|
| 应用产品层 | 业务功能 | Spring Boot 服务、Python AI 服务、前端应用 |
| 公共基座层 | WAF + 网关 + IAM | SafeLine、APISIX、Casdoor |
| 基础设施层 | 存储 / 缓存 / 搜索 | PostgreSQL、MinIO、Redis、Elasticsearch |

## 4 统一身份认证与网关

### 4.1 IAM —— Casdoor

- 全局用户目录、SSO 单点登录、JWT 令牌颁发
- 各子系统仅需信任并解析 JWT
- IAM 对接层做薄封装（Adapter / SPI），业务代码不直接调用 Casdoor 私有 API，便于未来按需切换至 Keycloak 等替代方案

### 4.2 API 网关 —— APISIX

系统唯一外部入口，接管所有 HTTP/gRPC 流量。

职责: JWT 校验、限流熔断、跨域处理、日志埋点。

双模式校验:
- 离线公钥校验 —— 面向高并发
- 实时穿透校验 —— 面向高敏感

**JWT 透传流程:**

```
             +-----------------+
             | Client (Web/App)|
             +--------+--------+
                      |
                      | 1. HTTPS 请求
                      v
             +--------+--------+
             |  SafeLine WAF   |
             | (恶意流量拦截)    |
             +--------+--------+
                      |
                      | 2. Bearer <JWT>
                      v
+-----------------------------------------------+
|               APISIX Gateway                  |
|                                               |
|  +----------------+    3. 获取公钥/校验签名     |     +---------------+
|  | JWT 鉴权 Plugin +---------------------------+---->|    Casdoor    |
|  +-------+--------+    4. 校验并返回结果        |<----|  (IAM 中心)   |
|          |                                    |     +---------------+
|          | 5. 提取 Token Payload              |
|          v                                    |
|  +----------------+                           |
|  | Header 转换重写  |                           |
|  | (X-User-ID etc)|                           |
|  +-------+--------+                           |
+-----------+-----------------------------------+
            |
            | 6. 转发 (携带身份 Header)
            v
     +------+-------+---------------+
     v              v               v
+---------+   +---------+    +---------+
| Spring  |   | Python  |    |  其他   |
| 业务服务 |   | AI 服务  |    | 扩展服务 |
+---------+   +---------+    +---------+
```

### 4.3 流量防护 —— SafeLine

部署在 APISIX 前的反向代理 WAF，所有外部流量先经 SafeLine 过滤再转发至 APISIX。

防护能力:

- OWASP Top 10（SQL 注入、XSS、CSRF、路径遍历等）
- CC 攻击防护 / 速率限制
- 爬虫识别与人机验证
- IP 黑白名单

部署方式: Docker Compose，与平台其他组件统一编排。

职责分离: SafeLine 负责恶意流量拦截，APISIX 负责业务路由与鉴权。

### 4.4 内部服务间通信

- 协议: gRPC (HTTP/2 + Protobuf)
- 同一内网微服务互信，不重复鉴权

### 4.5 网关路由配置

- 初期: 声明式 YAML (DB-less / Standalone Mode)
- 未来: etcd 动态路由 (见路线图)

### 4.6 服务发现

- 初期: Docker Compose 直连 / K8s CoreDNS
- 不引入额外服务发现组件

## 6 多租户数据隔离

- Casdoor 多租户 Realm，含 Client 配置和 RBAC 角色
- 网关层注入 `X-Tenant-ID`，下游服务据此隔离数据
- 方案: 共享数据库 + 租户字段过滤

## 7 安全架构

- WAF: SafeLine 前置拦截恶意流量 (OWASP Top 10 / CC / 爬虫)
- 认证: Casdoor 统一颁发 JWT，网关层校验
- 授权: RBAC，Casdoor 管理角色与权限
- 传输: HTTPS (外部)，内网 gRPC 互信
- 敏感数据: 环境变量 / Secret 管理，不硬编码

## 13 开发协作

### 13.1 API First

OpenAPI 3.x 规范，YAML 格式。各服务在自身仓库 `api/` 目录维护 spec，APISIX 统一暴露。

**工具链**

| 工具 | 职责 |
|---|---|
| Redocly CLI | Spec 多文件拆分 / 合并 / lint 校验（离线） |
| Swagger UI | API 文档可视化（自托管） |

**Spec 目录结构**

```
api/
  openapi.yaml          ← 入口，只含 info + servers + $ref
  paths/
    users.yaml
    tenants.yaml
  components/
    schemas/
    responses/
```

**工作流:** 编写 spec → Redocly lint 校验 → AI 生成代码 → Swagger UI 查阅文档

---

### 13.2 分形文档体系

每级目录拥有独立上下文，只需阅读「当前层 + 上一层」即可开工。

**仓库策略:** 各 Service 独立 Git 仓库，polaris-base 仅 `services/README.md` 作为索引。

```
polaris-base/                        <service-repo>/
  proto/              ← gRPC 契约        CLAUDE.md     ← AI 上下文
  services/README.md  ← 服务清单         README.md     ← 人类文档
  deploy/             ← 部署编排          api/openapi.yaml
  docs/               ← 平台文档          src/module/CLAUDE.md ← 按需
  CLAUDE.md           ← AI 全局约定
```

**文档分层**

| 层级 | README.md (面向人类) | CLAUDE.md (面向 AI) | 必须 |
|---|---|---|---|
| 平台根 | 项目介绍、启动方式 | 全局架构约定、跨服务禁忌 | 是 |
| 服务根 | 职责摘要 + 文档链接 | 服务级约定、禁忌、上下文指针 | 是 |
| 模块层 | 模块职责、关键入口 | 模块级约定与禁忌 | 按需 |

README 与 CLAUDE 不重复。README 写 What & How；CLAUDE 只写约定、禁忌、上下文指针。

**CLAUDE.md 四要素:** 本层职责 / 关键约定 / 禁止事项 / 上下文指针

> Claude Code 自动沿目录树向上加载所有 CLAUDE.md，无需手动引用。

### 13.3 AI 辅助开发

**上下文投喂原则:** 最小够用。

| 任务粒度 | 投喂文档 |
|---|---|
| 跨服务设计 | 平台顶层文档 |
| 单服务功能 | 服务 CLAUDE.md |
| 模块/函数 | 模块 README |
| 行级调试 | 当前文件上下文 |

**任务分解:** 接口契约先行 → 单服务隔离开发 → 增量验证（单测 / curl / gRPC）

**`.claude/` 目录:** 各服务仓库及 polaris-base 根目录下按需创建 `.claude/` 目录，存放自定义命令等 AI 协作配置。

# [PolarisNexus](https://github.com/PolarisNexus) 平台顶层技术栈

> 地基文档 —— 面向 AI 开发参考，不解释原理，原理另建专项文档。
>
> 旨在构建一个**基于多语言微服务（Polyglot Microservices）的企业级技术底座**。

---

## 1 架构概述

多语言微服务 + 可组合架构。用成熟开源组件拼装平台基座，业务以独立产品接入。

**一个平台基座 + 多产品矩阵**

- **平台基座:** 统一接管 IAM、网关、多租户路由、可观测性、容器运行环境。
- **产品接入:** Java 业务群、Python AI 群及其他语言模块，通过 OpenAPI 规范、HTTP/gRPC 协议对接，独立部署。

### 架构分层

| 层 | 职责 | 核心组件 |
|---|---|---|
| 应用产品层 | 业务功能 | Spring Boot 服务、Python AI 服务、前端应用 |
| 公共基座层 | WAF + 网关 + IAM + 管理控制台 | Coraza（APISIX 内置）、APISIX（etcd 模式）、Authentik、platform-admin |
| 基础设施层 | 存储 / 缓存 / 搜索 | PostgreSQL、MinIO、Redis、Elasticsearch |

## 4 统一身份认证与网关

### 4.1 IAM —— Authentik（ADR-0010）

- MIT 许可，自带现代 Admin UI，无商业品牌侵入
- 全局用户目录、SSO 单点登录（OIDC/OAuth2/SAML）、JWT 令牌颁发
- Flows 可视化认证流编排——拖拽式定义注册/登录/找回密码流程
- 各子系统仅需信任并解析 JWT
- IAM 对接层做薄封装（Adapter / SPI，ADR-0004），业务代码通过标准 OIDC 对接，不直接依赖 Authentik 私有 API

### 4.2 API 网关 —— APISIX（etcd 模式，ADR-0002）

系统唯一外部入口，接管所有 HTTP/gRPC 流量。

职责: JWT 校验、限流熔断、跨域处理、日志埋点、**AI 模型路由、WAF**。

部署模式: etcd 模式（传统模式），通过 Admin API 动态管理路由和插件。

**AI Gateway 能力**（APISIX 内置插件）:
- ai-proxy / ai-proxy-multi: 多 LLM 统一代理与负载均衡
- ai-rate-limiting: token 级限流（TPM 配额）
- ai-rag: RAG 增强生成

**JWT 透传流程:**

```
             +-----------------+
             | Client (Web/App)|
             +--------+--------+
                      |
                      | 1. HTTPS 请求
                      v
+-----------------------------------------------+
|            APISIX Gateway (etcd 模式)          |
|                                               |
|  +----------------+                           |
|  | Coraza WAF     | ← OWASP CRS 规则检查      |
|  +-------+--------+                           |
|          |                                    |
|  +----------------+    2. 获取公钥/校验签名     |     +---------------+
|  | JWT 鉴权 Plugin +---------------------------+---->|   Authentik   |
|  +-------+--------+    3. 校验并返回结果        |<----|  (IAM 中心)   |
|          |                                    |     +---------------+
|          | 4. 提取 Token Payload              |
|          v                                    |
|  +----------------+                           |
|  | Header 转换重写  |                           |
|  | (X-User-ID etc)|                           |
|  +-------+--------+                           |
+-----------+-----------------------------------+
            |
            | 5. 转发 (携带身份 Header)
            v
     +------+-------+---------------+
     v              v               v
+---------+   +---------+    +---------+
| Spring  |   | Python  |    |  Go    |
| 业务服务 |   | AI 服务  |    | 共享服务 |
+---------+   +---------+    +---------+
```

### 4.3 流量防护 —— Coraza WAF + CrowdSec（ADR-0012）

分层部署：

| 层 | 引擎 | 形态 |
|----|------|------|
| 请求层 WAF | **Coraza** + OWASP CRS | APISIX Wasm 插件，零容器 |
| 行为层 | **CrowdSec** CE | 独立容器（`components/crowdsec/`） |
| 人机验证 | **Cloudflare Turnstile** | SaaS |
| 设备指纹 | **FingerprintJS Pro** | SaaS |

覆盖能力：OWASP Top 10（CRS）、扫描器/爆破/CVE 批量利用（CrowdSec scenarios）、社区威胁情报黑名单、挑战验证、跨会话设备追踪。

### 4.4 平台管理控制台 —— platform-admin（ADR-0013）

自建 Go + React + AntD Pro 管理控制台。P1 覆盖 Gateway 路由只读浏览 / 插件与限流 PATCH、WAF 攻击日志查询（基于 ES）、CrowdSec 决策与告警管理。WAF 规则启停走 `components/apisix/routes/95-coraza.yaml` Git 源 PR（UI P2 再评估）。IAM 管理复用 Authentik 自带 Admin UI。

### 4.5 内部服务间通信

- 协议: gRPC (HTTP/2 + Protobuf)
- 同一内网微服务互信，不重复鉴权

### 4.6 网关路由配置

- etcd 模式，通过 Admin API 动态管理路由和插件（ADR-0002）
- **Git 源 ↔ etcd 同步**：结构配置（路由/Upstream/SSL、Coraza CRS、全局 logger）Git 源在 `components/apisix/routes/*.yaml`，CI 跑 `apisix-apply-routes.sh` 幂等写入 etcd；运行时策略（限流阈值、插件参数）走 platform-admin UI 直改

### 4.7 服务发现

- 初期: Docker Compose 直连 / K8s CoreDNS
- 不引入额外服务发现组件

## 7 安全架构

- WAF: Coraza（APISIX Wasm 插件）拦截恶意流量（OWASP CRS 规则集覆盖 Top 10）
- Bot 防护: 自建 Go 服务分析 access log + Cloudflare Turnstile 人机验证 + FingerprintJS 设备指纹
- 认证: Authentik 统一颁发 JWT，APISIX jwt-auth 插件校验
- 授权: RBAC，Authentik Groups + Policies 管理角色与权限
- 传输: HTTPS (外部)，内网 gRPC 互信
- 敏感数据: 环境变量 / Secret 管理，不硬编码

## 13 开发协作

### 13.1 API First

OpenAPI 3.x 规范，YAML 格式。各服务在自身仓库 `api/` 目录维护 spec，APISIX 统一暴露。

**工具链**

| 工具 | 职责 |
|---|---|
| Redocly CLI | Spec 多文件拆分 / 合并 / lint 校验（离线） |
| Swagger UI | API 文档可视化（自托管） |

**Spec 目录结构**

```
api/
  openapi.yaml          ← 入口，只含 info + servers + $ref
  paths/
    users.yaml
    tenants.yaml
  components/
    schemas/
    responses/
```

**工作流:** 编写 spec → Redocly lint 校验 → AI 生成代码 → Swagger UI 查阅文档

---

### 13.2 分形文档体系

每级目录拥有独立上下文，只需阅读「当前层 + 上一层」即可开工。

**仓库策略:** 各 Service 独立 Git 仓库，polaris-base 仅 `services/README.md` 作为索引。

```
polaris-base/                        <service-repo>/
  proto/              ← gRPC 契约        CLAUDE.md     ← AI 上下文
  services/README.md  ← 服务清单         README.md     ← 人类文档
  deploy/             ← 部署编排          api/openapi.yaml
  docs/               ← 平台文档          src/module/CLAUDE.md ← 按需
  CLAUDE.md           ← AI 全局约定
```

**文档分层**

| 层级 | README.md (面向人类) | CLAUDE.md (面向 AI) | 必须 |
|---|---|---|---|
| 平台根 | 项目介绍、启动方式 | 全局架构约定、跨服务禁忌 | 是 |
| 服务根 | 职责摘要 + 文档链接 | 服务级约定、禁忌、上下文指针 | 是 |
| 模块层 | 模块职责、关键入口 | 模块级约定与禁忌 | 按需 |

README 与 CLAUDE 不重复。README 写 What & How；CLAUDE 只写约定、禁忌、上下文指针。

**CLAUDE.md 四要素:** 本层职责 / 关键约定 / 禁止事项 / 上下文指针

> Claude Code 自动沿目录树向上加载所有 CLAUDE.md，无需手动引用。

### 13.3 AI 辅助开发

**上下文投喂原则:** 最小够用。

| 任务粒度 | 投喂文档 |
|---|---|
| 跨服务设计 | 平台顶层文档 |
| 单服务功能 | 服务 CLAUDE.md |
| 模块/函数 | 模块 README |
| 行级调试 | 当前文件上下文 |

**任务分解:** 接口契约先行 → 单服务隔离开发 → 增量验证（单测 / curl / gRPC）

**`.claude/` 目录:** 各服务仓库及 polaris-base 根目录下按需创建 `.claude/` 目录，存放自定义命令等 AI 协作配置。
